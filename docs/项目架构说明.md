# AgentGraphRAG 项目架构说明

## 1. 项目目标

AgentGraphRAG 面向航空质量问题溯因分析场景，核心目标不是做一个普通 RAG demo，而是把结构化问题记录、非结构化分析文档、图谱关系和历史案例统一起来，形成一个可检索、可追溯、可解释的多 Agent GraphRAG 系统。

系统需要解决四类核心问题：

- 如何把本地上传的 PDF / DOCX / 图片文档转成可用知识。
- 如何把结构化记录和非结构化知识同时装载进向量库和图数据库。
- 如何在问答阶段根据问题类型自动决定 graph / vector / hybrid 检索策略。
- 如何输出带证据和推理路径的答案，并保留审计 trace。

## 2. 总体架构

系统分成四层：

1. 前端工作台层
2. FastAPI 接口层
3. 离线知识构建层
4. 在线多 Agent 问答层

另外还有三条横切能力：

- 配置中心
- Prompt Registry
- 审计与风险控制

## 3. 前端工作台

工作台入口是 `/workspace`，由 [app/api/routes_workspace.py](D:/Project/AgentGraphRAG/app/api/routes_workspace.py) 返回纯前端 HTML 页面。

这个页面现在实现了五类能力：

- 从本地选择非结构化文件并上传。
- 上传成功后自动调用入库接口更新知识库。
- 查看文档详情和 chunk 预览。
- 在同一页面直接提问，并展示答案、证据、推理路径和完整 trace。
- 查看最近文档、最近入库任务、最近审计日志和图谱快照。

页面没有引入额外前端框架，原因是当前项目更适合保持单体可运行和部署简单。页面直接调用以下接口：

- `POST /api/v1/documents/upload`
- `POST /api/v1/ingestion/start`
- `POST /api/v1/qa/ask`
- `GET /api/v1/documents`
- `GET /api/v1/documents/{document_id}`
- `GET /api/v1/documents/{document_id}/chunks`
- `GET /api/v1/ingestion`
- `GET /api/v1/audit/logs`
- `GET /api/v1/audit/logs/{trace_id}`
- `GET /api/v1/knowledge/graph`

## 4. 配置中心

配置中心入口是 `/settings`，由 [app/core/settings_manager.py](D:/Project/AgentGraphRAG/app/core/settings_manager.py) 和 [app/core/settings_catalog.py](D:/Project/AgentGraphRAG/app/core/settings_catalog.py) 共同驱动。

### 4.1 配置设计

配置被分成几组：

- 共享 LLM 网关
- 问答模型配置
- 推理模型配置
- Embedding 配置
- 文档解析配置
- Neo4j 配置
- Milvus 配置
- MySQL 配置

### 4.2 为什么要拆成问答模型和推理模型

这个项目不是单模型问答。

- 问答模型负责最终自然语言回答生成。
- 推理模型负责问题解析、Cypher 规划、检索路由和证据校验。

所以配置里单独提供了：

- `answer_llm_model`
- `reasoning_llm_model`

这样你可以把 `Qwen2.5-3B-Instruct` 配成回答模型，把 `Qwen2.5-Coder-3B-Instruct` 配成推理模型，而不用改代码。

### 4.3 连接测试

配置页支持调用 `GET /api/v1/settings/test-connections` 做当前 MySQL、Milvus、Neo4j 的连通性测试。

## 5. Prompt Registry

Prompt Registry 由 [app/prompts/registry.py](D:/Project/AgentGraphRAG/app/prompts/registry.py) 管理，并通过 `GET /api/v1/prompts` 暴露。

当前 registry 记录了每个 prompt 的：

- 名称
- 版本
- 文件名
- 用途
- 文件路径

这样做的目的是把 prompt 工程从“散落的 txt 文件”提升成“可登记、可查询的模板资产”。

## 6. 离线知识构建层

离线阶段主要由以下模块组成：

- [app/offline/ingest_structured.py](D:/Project/AgentGraphRAG/app/offline/ingest_structured.py)
- [app/offline/ingest_unstructured.py](D:/Project/AgentGraphRAG/app/offline/ingest_unstructured.py)
- [app/offline/document_classifier.py](D:/Project/AgentGraphRAG/app/offline/document_classifier.py)
- [app/offline/chunking.py](D:/Project/AgentGraphRAG/app/offline/chunking.py)
- [app/offline/extraction.py](D:/Project/AgentGraphRAG/app/offline/extraction.py)
- [app/offline/loaders/milvus_loader.py](D:/Project/AgentGraphRAG/app/offline/loaders/milvus_loader.py)
- [app/offline/loaders/neo4j_loader.py](D:/Project/AgentGraphRAG/app/offline/loaders/neo4j_loader.py)
- [app/offline/loaders/case_loader.py](D:/Project/AgentGraphRAG/app/offline/loaders/case_loader.py)

### 6.1 结构化数据入库

结构化记录先做归一化，然后转成标准 chunk 文本，再同时写入：

- Milvus `knowledge_chunks`
- Neo4j 图谱

### 6.2 非结构化文档入库

本地上传文档后，处理链路是：

1. 物理解析
2. 文档分类
3. 分块
4. 实体关系抽取
5. Milvus 装载
6. Neo4j 装载
7. case memory 装载

### 6.3 文档分类

文档分类由 [app/offline/document_classifier.py](D:/Project/AgentGraphRAG/app/offline/document_classifier.py) 实现，当前支持三类：

- `issue_record`
- `analysis_report`
- `action_report`

分类逻辑目前仍以规则为主，保留 prompt 入口，后续可以替换成真实推理模型输出结构化分类结果。

### 6.4 分块

分块由 [app/offline/chunking.py](D:/Project/AgentGraphRAG/app/offline/chunking.py) 实现。

主要规则：

- issue record 直接转成结构化摘要 chunk
- analysis report 按 section/page 切块
- action report 按步骤切块
- 过长文本再按句子切分
- 自动打 `chunk_type`

### 6.5 实体关系抽取

抽取由 [app/offline/extraction.py](D:/Project/AgentGraphRAG/app/offline/extraction.py) 负责。

当前实现使用规则和别名归一的混合方式，输出统一 schema：

- `Entity`
- `Relation`

关系会映射到领域本体，比如：

- `CAUSES`
- `MITIGATES`
- `HANDLES`
- `HAS_PHENOMENON`
- `INVOLVES_COMPONENT`

### 6.6 案例记忆

每次文档入库后，会抽一个简化 case summary，写入：

- MySQL `cases`
- Milvus `case_memory`

这样 hybrid 检索就能把 chunk 和历史案例同时召回。

## 7. 存储层

### 7.1 MySQL

当前管理数据通过 [app/core/db_mysql.py](D:/Project/AgentGraphRAG/app/core/db_mysql.py) 做统一封装。

它承载：

- documents
- ingestion_jobs
- chunks
- cases
- audit_logs
- risk_rules

开发和测试环境允许先用 sqlite 连接串兼容，但长期目标是 MySQL。

### 7.2 Milvus

Milvus 封装在 [app/core/db_milvus.py](D:/Project/AgentGraphRAG/app/core/db_milvus.py)。

当前分成两类 collection：

- `knowledge_chunks`
- `case_memory`

这和 `Project.md` 中 chunk 向量索引与案例记忆索引分离的设计一致。

### 7.3 Neo4j

Neo4j 封装在 [app/core/db_neo4j.py](D:/Project/AgentGraphRAG/app/core/db_neo4j.py)。

它负责：

- 实体节点合并
- 关系边合并
- 图谱查询
- 多跳路径回显
- 图谱快照输出

## 8. 在线多 Agent 问答层

在线问答的主入口是 [app/online/workflow.py](D:/Project/AgentGraphRAG/app/online/workflow.py)。

核心 Agent 在 [app/online/agents.py](D:/Project/AgentGraphRAG/app/online/agents.py) 里定义。

### 8.1 Query Router Agent

对应模块：

- [app/online/question_parser.py](D:/Project/AgentGraphRAG/app/online/question_parser.py)
- [app/online/router.py](D:/Project/AgentGraphRAG/app/online/router.py)

职责：

- 解析问题
- 识别实体和关系意图
- 判断是否需要多跳
- 决定 `graph / vector / hybrid`

### 8.2 Retrieval Agent

对应模块：

- [app/online/query_builder.py](D:/Project/AgentGraphRAG/app/online/query_builder.py)
- [app/online/retrieve_vector.py](D:/Project/AgentGraphRAG/app/online/retrieve_vector.py)
- [app/online/retrieve_graph.py](D:/Project/AgentGraphRAG/app/online/retrieve_graph.py)
- [app/online/retrieve_hybrid.py](D:/Project/AgentGraphRAG/app/online/retrieve_hybrid.py)
- [app/online/rerank.py](D:/Project/AgentGraphRAG/app/online/rerank.py)

职责：

- 生成 query plan
- 执行 graph / vector / hybrid 检索
- 混合 case memory
- 对不同来源证据做统一重排

### 8.3 Graph Reasoning Agent

对应模块：

- [app/online/reasoning.py](D:/Project/AgentGraphRAG/app/online/reasoning.py)

职责：

- 把重排证据整理成 reasoning path
- 为最终回答提供可解释链路

### 8.4 Verification Agent

对应模块：

- [app/online/verifier.py](D:/Project/AgentGraphRAG/app/online/verifier.py)

职责：

- 评估证据是否充分
- 发现冲突
- 决定是否降级为 `summary_only` 或 `conflict_notice`

### 8.5 Answer Synthesis Agent

对应模块：

- [app/online/answer_generator.py](D:/Project/AgentGraphRAG/app/online/answer_generator.py)

职责：

- 组织最终答案
- 输出 evidence
- 输出 reasoning path
- 在没有真实 LLM 或 LLM 调用失败时回退到模板回答

## 9. LLM 与 Embedding 抽象层

### 9.1 LLM 抽象

[app/core/llm.py](D:/Project/AgentGraphRAG/app/core/llm.py) 提供统一的角色化模型访问。

支持：

- OpenAI
- Azure OpenAI
- DeepSeek
- DashScope 兼容模式
- OpenAI-compatible 网关
- Ollama OpenAI 兼容入口
- 规则回退模式

### 9.2 Embedding 抽象

[app/core/embeddings.py](D:/Project/AgentGraphRAG/app/core/embeddings.py) 支持：

- 本地占位向量
- DashScope 原生 embedding
- OpenAI-compatible embedding
- Ollama / 本地 embedding 网关

这样你可以把 `bge-m3` 通过本地兼容服务挂接进来，而不需要改主链路代码。

## 10. 审计与风险控制

### 10.1 风险控制

风险控制由：

- [app/risk/risk_classifier.py](D:/Project/AgentGraphRAG/app/risk/risk_classifier.py)
- [app/online/verifier.py](D:/Project/AgentGraphRAG/app/online/verifier.py)

共同完成。

当前策略包括：

- 问题风险初判
- 高风险下提高证据阈值
- 证据不足时降级输出
- 冲突时阻止强结论

### 10.2 审计日志

审计日志由 [app/risk/audit.py](D:/Project/AgentGraphRAG/app/risk/audit.py) 写入。

每次问答会记录：

- trace_id
- user_id
- question
- route
- risk_level
- evidence_ids
- fallback_mode
- agent_traces
- reasoning_path
- answer_payload
- final_answer
